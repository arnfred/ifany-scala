# Frontend Modernization Plan: HTMX + Tailwind + ScalaTags

## Goal

Replace the current frontend stack (Bootstrap 3, jQuery, AMD/curl.js, string-interpolated templates) with a modern, minimal-JS stack: **Tailwind CSS v4** for styling, **htmx** for interactivity, **ScalaTags** for type-safe HTML generation, and a small vanilla JS module for the lightbox overlay. Remove jQuery, Underscore, History.js, radio.js, curl.js, and all AMD infrastructure.

## Why This Stack

| Current | Proposed | Rationale |
|---------|----------|-----------|
| Bootstrap 3 grid + custom CSS | Tailwind CSS v4 | Utility-first, no dead CSS, responsive by default, no custom class naming |
| jQuery + Underscore + radio.js | htmx + vanilla JS | htmx handles all server interactions declaratively; vanilla JS handles the one client-only feature (lightbox) |
| curl.js AMD loader | `<script>` tags (ES module for lightbox) | No module bundler needed; htmx is a single `<script>`, lightbox is one small file |
| History.js | Native `history.replaceState` | History.js was a polyfill for IE; all modern browsers support the History API natively |
| String interpolation templates | ScalaTags DSL | Type-safe, composable, catches HTML errors at compile time, proper escaping |

## Design Principles

- **Minimal JavaScript**: The only JS should be the htmx library (~14KB) and a small lightbox module (~100 lines). Everything else is HTML attributes + server responses.
- **Compiled CSS only**: Use Tailwind standalone CLI to generate a minimal, purged CSS file. No CDN in production — users never load the full Tailwind runtime. Use CDN during development only.
- **Progressive migration**: Can be done page-by-page (frontpage, gallery, album) since each page type is a separate template.
- **Image performance — no regressions**: The current site has smart image loading behavior that must be preserved and improved:
  - Album images use `srcset` with all size variants + `loading="lazy"` — the browser handles responsive selection and lazy loading of nearby images as you scroll
  - The lightbox clones already-loaded `<img>` elements from the DOM (with their cached srcset), so opening an overlay is instant — no re-fetch
  - Frontpage thumbnails defer loading until the gallery accordion opens (prevents loading hundreds of thumbnails upfront)
  - Videos use `preload="none"` until scrolled into viewport
  - **New improvement**: The lightbox should prefetch the prev/next images when viewing an image, so arrow navigation feels instant even for images further down the page that haven't been lazy-loaded yet (see Lightbox Module section)
- **Server-rendered**: All content continues to be server-rendered. htmx is used only for interactive features (gallery accordion expand, future edit mode), not for initial page load.
- **DualRow equal-height via CSS flexbox**: The current Scala `DualRow` computes pixel-exact width percentages so two side-by-side images share equal height. This can be replaced with CSS flexbox — set `flex-grow` to each image's aspect ratio (width/height), and flexbox will distribute space so both images end up the same height automatically. See the "DualRow — Flexbox Equal Height" section for the proof and implementation.
- **Modern CSS layout**: Use CSS Grid and Flexbox (via Tailwind) for all page structure. No Bootstrap grid, no float-based layouts.
- **ScalaTags class conventions**:
  - Separate Tailwind classes with newlines when there are more than a few on an element
  - Factor out shared class strings into `val` variables to avoid duplication — e.g. `val albumRowCls = "..."` reused across cover rows and dual rows
  - Use helper methods for repeated patterns (navigation, image boxes)

## ScalaTags Template System

### Why ScalaTags over keeping string interpolation

1. **Type safety**: Misspelled tag/attribute names caught at compile time
2. **Auto-escaping**: User-provided text (titles, descriptions) is automatically HTML-escaped
3. **Composability**: Template fragments are typed values, not strings — can be passed, transformed, collected
4. **Custom attributes**: `attr("hx-get") := "/api/..."` works naturally for htmx integration
5. **No runtime errors**: Can't produce malformed HTML through string concatenation mistakes

### How it replaces the current Template trait

Current pattern:

## Tailwind CSS Strategy

### Development: CDN

During development, use the Tailwind v4 browser CDN. This requires zero setup — just a `<script>` tag. It scans the DOM for Tailwind classes and generates styles on the fly.

```html
<script src="https://cdn.jsdelivr.net/npm/@tailwindcss/browser@4"></script>
```

Limitation: not suitable for production (adds ~100ms page load, no purging).

### Production: Standalone CLI (Required)

For production, always use the Tailwind standalone CLI to generate a minimal, purged CSS file. No CDN runtime in production — users get a small, static CSS file.

1. Download the standalone binary for Linux (the Dokku server)
2. Create `resources/css/input.css` with `@import "tailwindcss"` and the `@theme` block (see Color Palette section)
3. Run: `tailwindcss -i resources/css/input.css -o resources/css/site.css --minify`
4. Reference `site.css` via `<link>` instead of the CDN script tag
5. Add as a build step — either a Dokku pre-deploy script or a Makefile target run before `git push`

The compiled CSS file is committed to the repo so the Dokku build doesn't need to install the Tailwind CLI. Re-run the CLI locally whenever templates change, similar to how you'd recompile any asset.

```bash
# One-time setup: download Tailwind standalone CLI
curl -sLO https://github.com/tailwindlabs/tailwindcss/releases/latest/download/tailwindcss-macos-arm64
chmod +x tailwindcss-macos-arm64
mv tailwindcss-macos-arm64 /usr/local/bin/tailwindcss

# Build CSS (run after template changes)
tailwindcss -i resources/css/input.css -o resources/css/site.css --minify

# Watch mode during development (alternative to CDN)
tailwindcss -i resources/css/input.css -o resources/css/site.css --watch
```

## htmx Usage

htmx replaces the jQuery-driven interactive behaviors. The key principle: **htmx is for server interactions, not client-side-only behavior**. The lightbox (which is entirely client-side) stays as vanilla JS.

### Gallery Accordion (Frontpage)

Currently: jQuery click handler toggles `.album-hidden` divs, loads lazy thumbnails by swapping `src` from `href`.

With htmx: The album list for each gallery is loaded on demand from the server when the gallery is clicked.

```scala
// Gallery header (clickable)
div(cls := "...",
  hxGet := s"/api/gallery/${g.url}/albums",
  hxTarget := s"#albums-${g.url}",
  hxSwap := "innerHTML",
  hxTrigger := "click",
  // Gallery name, cover image, metadata...
)

// Placeholder for album list
div(id := s"albums-${g.url}")
```

New server endpoint `GET /api/gallery/:url/albums` returns an HTML fragment with the album list (thumbnails already loaded — no lazy `src` swapping needed since images are loaded on demand).

**Trade-off**: This adds a server round-trip when clicking a gallery. But it means the frontpage initial HTML is much smaller (no hidden album lists for all galleries), and thumbnails use real presigned URLs instead of the current lazy-swap hack.

### Preferred: `<details>/<summary>` with Deferred Thumbnail Loading

This is preferable over htmx because it's instant (no server round-trip). Use native `<details>/<summary>` HTML for the accordion, with deferred thumbnail loading to avoid loading hundreds of images upfront.

```html
<details class="group">
  <summary class="cursor-pointer list-none">
    <!-- Gallery header: cover image, name, metadata -->
  </summary>
  <!-- Album list with thumbnails -->
  <div class="album-list">
    <img loading="lazy" src="..." />
  </div>
</details>
```

**Deferred thumbnail loading**: Album thumbnails inside `<details>` use `loading="lazy"`. Since the content starts hidden (collapsed), the browser won't load these images until the `<details>` is opened *and* the images are near the viewport. This preserves the current behavior where thumbnails only load when the gallery is expanded.

**Animations**: The native `<details>` element doesn't animate by default, but Tailwind v4 can handle this with the `open:` variant and transition utilities:

```scala
tag("details")(cls := "group",
  tag("summary")(cls := "cursor-pointer list-none", /* gallery header */),
  div(cls := Seq(
    "grid",
    "grid-rows-[0fr]",
    "group-open:grid-rows-[1fr]",
    "transition-[grid-template-rows]",
    "duration-300",
    "ease-in-out",
    "overflow-hidden"
  ).mkString(" "),
    div(cls := "overflow-hidden",
      /* album list content */
    )
  )
)
```

The `grid-rows-[0fr]` → `group-open:grid-rows-[1fr]` transition with `transition-[grid-template-rows]` gives a smooth height animation matching the current jQuery `fadeToggle("fast")` behavior, using only Tailwind classes. The `group` / `group-open:` pattern uses Tailwind's group variant to detect the open state on the parent `<details>` element.

**Scroll-to behavior**: The current JS does `$.scrollTo(category, 400)` when expanding. Replace with a tiny inline script or CSS `scroll-margin-top` + `scrollIntoView()`:

```html
<details ontoggle="if(this.open) this.scrollIntoView({behavior:'smooth', block:'start'})">
```

### Where htmx is NOT Used

- **Lightbox overlay**: Pure client-side DOM manipulation. No server round-trip needed — the image data is already in the page.
- **Video play on hover**: Pure client-side. Use `onmouseover`/`onmouseleave` attributes or a tiny JS snippet.
- **Header font scaling**: Replace with CSS `clamp()` — no JS at all.

## Lightbox Module (Vanilla JS)

The lightbox is the one piece that requires JavaScript. It's rewritten as a single ES module (~100 lines), no dependencies.

### `resources/js/lightbox.js`

```javascript
// Lightbox module — vanilla JS, no dependencies
// Manages full-screen image overlay with keyboard/click navigation

const lightbox = {
  overlay: null,
  images: [],
  currentIndex: -1,
  albumUrl: '',

  init(albumData) {
    this.images = albumData.images
    this.albumUrl = albumData.url
    this.overlay = document.getElementById('overlay')
    this.bindEvents()
    this.openFromURL()
  },

  bindEvents() {
    // Click on any .media image → open overlay
    document.querySelectorAll('img.media, video.media').forEach((el, i) => {
      el.addEventListener('click', () => this.open(el.getAttribute('file')))
    })

    // Keyboard navigation
    document.addEventListener('keydown', (e) => {
      if (!this.isOpen()) return
      if (e.key === 'ArrowLeft' || e.key === 'Backspace') this.prev()
      else if (e.key === 'ArrowRight' || e.key === ' ' || e.key === 'Enter') this.next()
      else if (e.key === 'Escape') this.close()
    })

    // Click overlay background to close
    document.getElementById('overlay-img').addEventListener('click', () => this.close())
    document.getElementById('overlay-prev').addEventListener('click', () => this.prev())
    document.getElementById('overlay-next').addEventListener('click', () => this.next())
  },

  open(filename) {
    const index = this.images.findIndex(img => img.file === filename)
    if (index === -1) return
    this.currentIndex = index
    this.render()
    this.overlay.style.display = 'flex'
    document.body.style.overflowY = 'hidden'
    history.replaceState({}, '', this.baseURL() + filename)
  },

  close() {
    this.overlay.style.display = 'none'
    document.body.style.overflowY = 'auto'
    this.currentIndex = -1
    history.replaceState({}, '', this.baseURL())
  },

  next() { if (this.currentIndex < this.images.length - 1) this.open(this.images[this.currentIndex + 1].file) },
  prev() { if (this.currentIndex > 0) this.open(this.images[this.currentIndex - 1].file) },
  isOpen() { return this.currentIndex >= 0 },

  render() {
    const img = this.images[this.currentIndex]
    const container = document.getElementById('overlay-center-box')
    const sourceEl = document.getElementById('id-' + img.file.replaceAll('/', '--').replaceAll('.', '--'))
    const clone = sourceEl.cloneNode(true)
    // Remove existing media, insert clone
    container.querySelector('.media')?.remove()
    container.prepend(clone)
    document.getElementById('caption').textContent = img.description || ''
    // Show/hide nav arrows
    document.getElementById('overlay-prev').style.display = this.currentIndex > 0 ? '' : 'none'
    document.getElementById('overlay-next').style.display = this.currentIndex < this.images.length - 1 ? '' : 'none'
    this.resizeOverlay()
    // Prefetch prev/next images so arrow navigation feels instant
    this.prefetchAdjacent()
  },

  // Prefetch prev and next images by triggering browser cache via hidden Image objects.
  // This ensures that when the user presses an arrow key, the next image is already
  // in the browser cache even if it hasn't been lazy-loaded in the page yet
  // (e.g. images far down the page that the user hasn't scrolled to).
  prefetchAdjacent() {
    for (const offset of [-1, 1]) {
      const idx = this.currentIndex + offset
      if (idx < 0 || idx >= this.images.length) continue
      const adjImg = this.images[idx]
      if (adjImg.is_video) continue // don't prefetch videos
      const el = document.getElementById('id-' + adjImg.file.replaceAll('/', '--').replaceAll('.', '--'))
      if (el && el.complete) continue // already loaded
      // Create a hidden Image to trigger the browser to fetch the large variant
      const prefetch = new Image()
      prefetch.srcset = el ? el.getAttribute('srcset') || '' : ''
      prefetch.sizes = '83vw' // overlay is ~83% viewport width
      if (!prefetch.srcset && el) prefetch.src = el.src
    }
  },

  resizeOverlay() {
    const media = this.overlay.querySelector('#overlay-center-box .media')
    if (!media) return
    const onReady = () => {
      const w = media.naturalWidth || media.videoWidth
      const h = media.naturalHeight || media.videoHeight
      if (!w || !h) return
      const vpRatio = (window.innerWidth * 10 / 12) / window.innerHeight
      const imgRatio = w / h
      const pct = imgRatio >= vpRatio ? '100%' : (imgRatio / vpRatio * 100) + '%'
      media.style.width = pct
    }
    media.addEventListener('load', onReady, { once: true })
    media.addEventListener('loadedmetadata', onReady, { once: true })
    if (media.complete) onReady()
  },

  baseURL() {
    const parts = window.location.pathname.split('/')
    // URL pattern: /gallery/album/ or /gallery/album/image
    return '/' + parts[1] + '/' + parts[2] + '/'
  },

  openFromURL() {
    const parts = window.location.pathname.split('/').filter(Boolean)
    if (parts.length >= 3) {
      const filename = parts.slice(2).join('/')
      if (this.images.some(img => img.file === filename)) {
        this.open(filename)
      }
    }
  }
}

export default lightbox
```

### Video Behavior

Two behaviors to preserve: (1) lazy preloading when scrolled into view, (2) play on hover.

**Play on hover — HTML attributes only, no JS**:

The `onmouseover` and `onmouseleave` HTML attributes can be set directly in ScalaTags without any separate script:

```scala
video(
  cls := "media w-full",
  attr("muted") := "muted",
  attr("loop") := "loop",
  attr("controls") := "controls",
  playsinline := "true",
  preload := "none",
  poster := image.imageURL(view.album.url, "800"),
  attr("onmouseover") := "this.play()",
  attr("onmouseleave") := "this.pause()",
  tag("source")(src := image.videoURL(view.album.url), `type` := "video/mp4")
)
```

**Lazy preloading — requires minimal JS**:

Unfortunately, there's no CSS-only way to change `preload` from `none` to `auto` when an element enters the viewport. The `loading="lazy"` attribute only works on `<img>` and `<iframe>`, not `<video>`. So we need a small script for this. However, rather than an inline `<script>` block, add this to the lightbox module (it's album-page JS that's already loaded):

```javascript
// In lightbox.js init():
if ('IntersectionObserver' in window) {
  const observer = new IntersectionObserver((entries) => {
    entries.forEach(e => {
      if (e.isIntersecting) {
        e.target.preload = 'auto'
        observer.unobserve(e.target) // only need to trigger once
      }
    })
  })
  document.querySelectorAll('video').forEach(v => observer.observe(v))
}
```

This keeps all JS in one file (lightbox.js) rather than scattered inline scripts.

## Header Font Scaling — Pure CSS

The current JS measures the container width and dynamically adjusts font-size so the "if any" text fills its column. The original workaround was likely because early browsers had poor support for viewport-relative sizing, but modern CSS handles this well.

**First approach — try a large font and let it fit naturally**:

The header is in a `~25%` width column. Set a large viewport-relative font size and let the text fill the space:

```scala
h1(
  cls := Seq(
    "text-[min(7rem,18vw)]",  // large but capped — scales with viewport
    "sm:text-[min(7rem,12vw)]", // wider column on sm+, less aggressive scaling
    "font-bold",
    "leading-none",
    "tracking-tight",
    "w-fit"
  ).mkString("\n"),
  span(cls := "text-site-muted", "if"),
  span(cls := "text-site-secondary", "any")
)
```

The `min(7rem, 18vw)` caps the font at 7rem (matching the current max) but scales down with viewport width. `w-fit` ensures the element only takes the width of its content.

**If that doesn't perfectly fill the container**: Use `container` queries (well-supported in 2026) to scale relative to the parent column width instead of the viewport:

```css
/* In input.css */
@container sidebar (min-width: 0px) {
  .header-title { font-size: clamp(3rem, 50cqw, 7rem); }
}
```

```scala
div(cls := "@container/sidebar", /* parent column */
  h1(cls := "header-title font-bold leading-none tracking-tight",
    span(cls := "text-site-muted", "if"),
    span(cls := "text-site-secondary", "any")
  )
)
```

`50cqw` means 50% of the container's query width — the text will always fill roughly the same proportion of its parent column. No JavaScript needed either way.

### DualRow — Flexbox Equal Height

The current Scala `DualRow` class computes exact width percentages so two side-by-side images have equal height. This math can be replaced by CSS flexbox with `flex-grow` set to each image's aspect ratio.

**Why this works**: If image1 has aspect ratio `a1 = w1/h1` and image2 has `a2 = w2/h2`, and we set `flex-grow: a1` and `flex-grow: a2`:
- Flexbox distributes width proportionally: `Width1 = a1/(a1+a2) * total`, `Width2 = a2/(a1+a2) * total`
- Since each image fills its container width: `Height1 = Width1/a1 = total/(a1+a2)`, `Height2 = Width2/a2 = total/(a1+a2)`
- Therefore `Height1 = Height2` — equal heights, guaranteed by the flexbox algorithm.

This is mathematically identical to the current `DualRow` Scala computation but expressed in CSS.

```scala
// Shared class strings
val albumRowCls = "max-w-5xl mx-auto px-4 mb-px"
val imgCls = "w-full cursor-pointer"

def twoImageRow(view: AlbumView, row: DualRow): Frag = {
  val a1 = row.left.size(0).toDouble / row.left.size(1).toDouble   // aspect ratio
  val a2 = row.right.size(0).toDouble / row.right.size(1).toDouble

  frag(
    // Mobile: stacked (each full width)
    div(cls := s"$albumRowCls sm:hidden",
      imageBox(view, row.left, 100.0)
    ),
    div(cls := s"$albumRowCls sm:hidden",
      imageBox(view, row.right, 100.0)
    ),
    // Desktop: side by side with flex-grow = aspect ratio
    div(cls := s"$albumRowCls hidden sm:flex gap-px",
      div(
        style := s"flex-grow:$a1",
        imageBox(view, row.left, a1 / (a1 + a2) * 100)
      ),
      div(
        style := s"flex-grow:$a2",
        imageBox(view, row.right, a2 / (a1 + a2) * 100)
      )
    )
  )
}
```

**Note**: The `DualRow` case class and its Scala-side ratio computation can be simplified — just store the two images and let the template compute `flex-grow` from `image.size`. The view layer (`AlbumView.getRows`) still decides which images pair up; only the width math moves to CSS.

## Risks and Mitigations

| Risk | Mitigation |
|------|-----------|
| ScalaTags learning curve | DSL is small; most of it maps directly to HTML tag names |
| Compiled CSS out of sync with templates | Run `tailwindcss --watch` during dev; commit `site.css` before deploy |
| Overlay resize behavior may differ | Test carefully; the resize logic is preserved in vanilla JS |
| Deep linking to images may break | Lightbox module preserves the URL structure and `history.replaceState` behavior |
| `raw()` for user HTML content | Only use `raw()` for trusted content (descriptions with HTML links); all other text auto-escaped by ScalaTags |
| Lightbox arrow navigation slow for far-away images | `prefetchAdjacent()` creates hidden `Image` objects to trigger browser cache for prev/next images |
| `<details>` accordion animation cross-browser | CSS `grid-template-rows` animation works in all modern browsers; test on Safari |
| Frontpage thumbnail loading regression | Album thumbnails inside collapsed `<details>` use `loading="lazy"` — browser defers loading until opened |

## Relationship to edit.md

The edit mode planned in `edit.md` benefits significantly from this migration:

1. **htmx** handles all edit-mode server interactions (save, delete, create) without jQuery AJAX
2. **ScalaTags** makes it easy to conditionally render edit buttons when session is admin
3. **Tailwind** makes styling edit-mode UI (dropzones, toggles, progress bars) much simpler
4. The upload pipeline (client-side image resize, presigned S3 PUT) remains plain JS and is unaffected by this migration

Recommend completing this frontend modernization **before** implementing edit.md, as it provides the foundation that edit mode builds on.
