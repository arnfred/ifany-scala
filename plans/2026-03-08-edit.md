# In-Place Album & Gallery Editing Plan

## Goal

Replace the separate photo-tool (Python/Flask) with in-place editing built into ifany.org itself. Admins see edit buttons on album and gallery pages, enabling inline editing of titles, descriptions, captions, cover/banner flags, image management, and gallery organization — all without leaving the site.

## Design Principles

- **Admin-only**: All edit functionality gated behind the `Admin` role (existing auth)
- **htmx for server interactions**: Save, delete, create, and cache operations use htmx attributes — no manual fetch/XHR code
- **Client-side image processing**: Resize images in the browser using Canvas API before uploading — avoids large file transfers to the server and keeps the server lightweight
- **Chunked/individual uploads**: Upload one resized image at a time to avoid request size limits
- **Progressive enhancement**: Edit mode is conditionally rendered by ScalaTags templates when session is Admin — edit UI is part of the page, toggled with a CSS class
- **Tailwind styling**: Edit mode UI uses the existing Tailwind setup — no separate CSS file
- **Same DynamoDB schema**: No schema changes — maintain compatibility with existing data

## Architecture Overview

```
Browser (Admin)                        Server (Scala)                    AWS
─────────────────                      ──────────────                    ───
Click "Edit" →
Toggle edit mode (JS class toggle) →
Edit fields inline (contenteditable) →

Upload photos:
  1. Read file
  2. Extract EXIF datetime
  3. Resize to 9 sizes + original
     (Canvas API, quality 92)
  4. PUT each size as JPEG         →   (presigned URL)               →   S3 bucket

Save album metadata:
  hx-post="/api/album/:id/save"   →   Write to DynamoDB             →   Albums table
  Server returns HTML status fragment

Delete image:
  hx-delete="/api/album/:id/       →   Remove from DynamoDB          →   Albums table
    image/:file"                       Delete from S3                →   S3 bucket
  htmx removes the image element

Save gallery metadata:
  hx-post="/api/gallery/:id/save"  →   Write to DynamoDB             →   Galleries table
```

### Why client-side resizing?

The photo-tool currently uploads full-resolution images to the Flask server, which resizes them with Pillow and uploads to S3. This is problematic because:
1. Large file uploads (10-40MB RAW/JPEG) can timeout or fail
2. Server needs image processing libraries (not available in the Scala stack)
3. Processing 9 resize variants per image is CPU-intensive

Instead: the browser resizes images using the Canvas API, producing 10 JPEGs per photo (9 sizes + original). Each is uploaded individually via a presigned S3 URL that the server generates. The server never touches image bytes — it only provides presigned URLs and saves metadata to DynamoDB.

## Image Resize Specifications

Same dimensions as photo-tool, matching what the frontend expects:

| Label | Dimensions | Notes |
|-------|-----------|-------|
| 150 | 150x150 | Square crop from center |
| 400 | 400x300 | |
| 600 | 600x450 | |
| 800 | 800x600 | Primary display size |
| 1024 | 1024x768 | |
| 1280 | 1280x980 | |
| 1600 | 1600x1200 | |
| 2000 | 2000x1500 | |
| 3200 | 3200x2400 | |
| original | Native size | |

**Resize algorithm**: `Image.thumbnail()` equivalent — scale down to fit within bounds, maintaining aspect ratio. For 150x150, crop from center to square first, then resize. Skip sizes larger than the original. Encode as JPEG quality 92.

**S3 key format**: `albums/{album_url}/{filename}_{WxH}.jpg` and `albums/{album_url}/{filename}_original.jpg`

**Video**: Videos are uploaded as-is (MP4) to `albums/{album_url}/{filename}.mp4`. A poster frame is extracted client-side and uploaded as the image variants. Video resize/transcode is out of scope for v1 — just upload the MP4 directly via presigned URL.

## API Endpoints (New)

All require `Admin` session cookie. All routes live inside `GalleryPlan` (before the catch-all patterns) because Unfiltered Netty doesn't reliably chain async plan handlers.

API endpoints return HTML fragments for htmx to swap, not JSON (except presign which returns JSON for the upload JS).

### Album APIs

```
GET  /api/album/:albumUrl/presign?file=name&sizes=150,400,...,original
     → Returns JSON map of { "150": "https://s3...presigned-put-url", ... }
     Server generates S3 presigned PUT URLs for each size variant.

POST /api/album/:albumUrl/save
     Body: JSON album metadata (title, description, url, galleries, public, secret, images[])
     → Writes to DynamoDB. Returns HTML status fragment for htmx swap.

POST /api/album/create
     Body: JSON { url, title, galleries }
     → Creates empty album in DynamoDB. Returns redirect header.

DELETE /api/album/:albumUrl/image/:filename
     → Removes image from album metadata in DynamoDB.
       Deletes S3 objects (all size variants).
       Returns empty response (htmx removes the element via hx-swap="delete").
```

### Gallery APIs

```
POST /api/gallery/:galleryUrl/save
     Body: JSON { name, description }
     → Writes to DynamoDB. Returns HTML status fragment.

POST /api/gallery/create
     Body: JSON { url, name, description }
     → Creates gallery in DynamoDB. Returns redirect header.
```

### Cache Invalidation

```
POST /api/cache/clear
     → Clears in-memory album/gallery caches so changes are reflected.
       (Equivalent to existing /update route but as an API call.)
```

## htmx Integration

htmx is already loaded on every page (`Base.page` includes the CDN script). Edit mode uses htmx attributes directly in the ScalaTags templates for all server interactions.

### Save album metadata

```scala
// In AlbumTemplate, rendered only when session.exists(_.isAdmin):
button(
  cls := "bg-site-text text-site-bg px-4 py-2 text-sm hover:opacity-80",
  hxPost := s"/api/album/${view.album.url}/save",
  hxTarget := "#save-status",
  hxSwap := "innerHTML",
  hxVals := "js:{...collectAlbumEdits()}",
  hxIndicator := "#save-spinner",
  "Save"
)
span(id := "save-status", cls := "text-sm text-site-muted ml-2")
```

### Delete image

```scala
// Per-image delete button in edit mode:
button(
  cls := "absolute top-1 right-1 bg-red-600 text-white rounded-full w-6 h-6 text-xs hidden group-[.editing]:block",
  hxDelete := s"/api/album/${view.album.url}/image/${image.file}",
  hxTarget := s"#row-${image.id}",
  hxSwap := "delete",
  hxConfirm := "Delete this image?",
  raw("&times;")
)
```

### Save gallery metadata


### Where htmx is NOT used

- **Image upload pipeline**: Requires Canvas API resizing and multiple sequential S3 PUTs with progress tracking — this is vanilla JS in `edit.js`
- **Edit mode toggle**: Pure CSS class toggle (`.editing` on a parent element)
- **Cover/banner/published toggles**: Local state changes collected by JS, sent with the Save action

## Files to Create

### 1. `resources/js/edit.js` — Edit mode controller (vanilla JS ES module)

Single file handling all edit mode functionality. Loaded via `<script type="module">` only on pages where session is Admin.

Responsibilities:
- Toggle edit mode on/off (adds/removes `.editing` class on container)
- Make title, description, caption fields `contenteditable`
- Collect edits into JSON for `hx-vals` on save
- Show upload dropzone for new images
- Image upload pipeline: read → EXIF → resize → presign → PUT to S3 → progress UI
- Cover/banner/published toggle state management
- Report progress per file and per size variant

```javascript
// edit.js — vanilla JS, no dependencies (except htmx on page)
const edit = {
  editing: false,

  init(albumData) {
    this.albumUrl = albumData.url
    document.getElementById('edit-toggle')?.addEventListener('click', () => this.toggle())
    this.initDropzone()
  },

  toggle() {
    this.editing = !this.editing
    document.getElementById('album-container').classList.toggle('editing')
    // contenteditable on title, description, captions
    document.querySelectorAll('[data-editable]').forEach(el => {
      el.contentEditable = this.editing
    })
  },

  // ... upload pipeline, EXIF extraction, resize, progress
}
export default edit
```

### 2. EXIF extraction

Inline in `edit.js` or a small helper function. Only needs to extract `DateTimeOriginal` (tag 0x9003) from JPEG EXIF. ~50 lines.

### 3. No separate CSS file

Edit mode styles use Tailwind utilities directly in the ScalaTags templates:
- Edit buttons: `hidden group-[.editing]:block` (shown only in edit mode)
- Editable fields: `group-[.editing]:outline group-[.editing]:outline-site-link`
- Dropzone: `border-2 border-dashed border-site-muted p-8 text-center`
- Progress bars: `bg-site-link h-1` with width set by JS
- Image action buttons: positioned with `absolute` inside `relative` image containers

The `.editing` class is toggled on a parent `group` element, and Tailwind's `group-[.editing]:` variant controls visibility.

## Upload UX Flow

### Adding photos to an album

1. Admin clicks "Edit" on an album page → `.editing` class added, edit UI revealed
2. A dropzone appears: "Drop images here or click to select"
3. Admin selects/drops multiple JPEG files
4. For each file, a thumbnail preview appears with a progress bar
5. Browser reads EXIF date, resizes to all applicable dimensions
6. Browser requests presigned PUT URLs from server (`GET /api/album/:id/presign`)
7. Browser uploads each size variant directly to S3
8. Progress bar updates per-variant (e.g., "3/10 sizes uploaded")
9. Once all variants uploaded, image appears in the album grid
10. Admin can edit caption, toggle cover/banner/published
11. Admin clicks "Save" → htmx POSTs metadata to server → DynamoDB updated → status fragment swapped in

### Creating a new album

1. Admin visits a gallery page and clicks "New Album"
2. A form appears: album URL (slug), title
3. On submit → `hx-post="/api/album/create"` with `hx-vals`
4. Server creates album, returns redirect → htmx follows redirect to new album page

### Creating a new gallery

1. Admin visits frontpage and clicks "New Gallery"
2. A form appears: gallery URL (slug), name, description
3. On submit → `hx-post="/api/gallery/create"` with `hx-vals`
4. Server creates gallery, returns redirect

### Setting cover/banner images

In edit mode, each image has small toggle buttons (visible via `group-[.editing]:block`):
- Star icon → toggle `cover` (featured image, shown on frontpage)
- Flag icon → toggle `banner` (full-width display in album)
- Eye icon → toggle `published` (visibility)

Currently-active flags are highlighted. Click to toggle local state. Changes are collected by `edit.js` and sent with the "Save" action via `hx-vals`.

## Security Considerations

- All API endpoints check for Admin session cookie
- Presigned PUT URLs are scoped to specific S3 keys and expire in 15 minutes
- Content-Type on presigned URLs should be restricted to `image/jpeg` and `video/mp4`
- Album URL slugs should be validated (alphanumeric + hyphens only)
- JSON request bodies should be size-limited by Netty configuration
- CSRF: session cookie is SameSite=Lax, API endpoints only accept POST/DELETE — adequate for this use case
- htmx requests include `HX-Request: true` header which can be checked server-side

## Out of Scope (v1)

- Image reordering by drag-and-drop (can be added later)
- Bulk operations across albums
- Video transcoding (upload MP4 as-is)
- Undo/redo
- Collaborative editing
- Image cropping UI
- Converting old .conf format files
